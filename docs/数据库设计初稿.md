# 数据库设计初稿

## 一、设计说明

数据库使用 MySQL，主要保存用户信息、情绪日记、AI 对话记录、敏感词和用户反馈等数据。

本阶段为数据库设计初稿，优先保证结构清晰、字段够用，后续编码时可以根据实际需求调整字段长度、索引和约束。

## 二、命名规范

- 表名使用小写字母和下划线。
- 字段名使用小写字母和下划线。
- 主键统一使用 `id`。
- 时间字段统一使用 `create_time` 和 `update_time`。
- 用户相关数据通过 `user_id` 与用户表关联。

## 三、核心表设计

### 1. 用户表 user

用于保存用户账号、密码、角色和状态信息。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 用户 ID，主键 |
| username | varchar(50) | 用户名，唯一 |
| password | varchar(255) | 加密后的密码 |
| nickname | varchar(50) | 用户昵称 |
| avatar | varchar(255) | 头像地址 |
| email | varchar(100) | 邮箱 |
| role | varchar(20) | 用户角色，user 或 admin |
| status | int | 状态，1 正常，0 禁用 |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |

建议索引：

- `username` 唯一索引
- `role` 普通索引

### 2. 情绪日记表 emotion_diary

用于保存用户每日情绪日记、情绪分析结果和 AI 建议。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 日记 ID，主键 |
| user_id | bigint | 用户 ID |
| title | varchar(100) | 日记标题 |
| content | text | 日记内容 |
| mood_tag | varchar(50) | 用户自选心情标签 |
| emotion_polarity | varchar(20) | 情绪极性，positive、negative、neutral |
| emotion_type | varchar(50) | 情绪类型，如开心、焦虑、悲伤、疲惫 |
| emotion_score | int | 情绪强度，1 到 5 |
| keywords | varchar(255) | 关键词，多个关键词用逗号分隔 |
| ai_summary | text | AI 或 NLP 分析摘要 |
| ai_suggestion | text | AI 生成的情绪建议 |
| risk_flag | int | 是否存在风险表达，1 是，0 否 |
| diary_date | date | 日记日期 |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |

建议索引：

- `user_id` 普通索引
- `diary_date` 普通索引
- `emotion_type` 普通索引

说明：

- 日记内容属于隐私数据，接口层必须限制用户只能访问自己的数据。
- 情绪统计可以优先从该表实时查询，初期不一定单独建统计表。

### 3. AI 对话消息表 ai_chat_message

用于保存用户与 AI 陪伴智能体的对话记录。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 消息 ID，主键 |
| user_id | bigint | 用户 ID |
| role | varchar(20) | 消息角色，user 或 assistant |
| content | text | 消息内容 |
| emotion_type | varchar(50) | 当前消息识别出的情绪类型 |
| risk_flag | int | 是否存在风险表达，1 是，0 否 |
| create_time | datetime | 创建时间 |

建议索引：

- `user_id` 普通索引
- `create_time` 普通索引

说明：

- 一次用户提问和一次 AI 回复可以分别保存为两条记录。
- 后续生成对话上下文时，可以查询用户最近若干条消息。

### 4. 敏感词表 sensitive_word

用于保存系统风险词、敏感词和提示词规则。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 敏感词 ID，主键 |
| word | varchar(100) | 敏感词内容 |
| type | varchar(50) | 类型，如 risk、negative、custom |
| level | int | 风险等级，1 到 5 |
| status | int | 状态，1 启用，0 禁用 |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |

建议索引：

- `word` 普通索引
- `type` 普通索引

说明：

- 风险词检测用于触发安全提示。
- 本系统只做基础风险提醒，不做心理诊断。

### 5. 用户反馈表 feedback

用于保存用户对系统的意见和问题反馈。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 反馈 ID，主键 |
| user_id | bigint | 用户 ID |
| content | text | 反馈内容 |
| status | varchar(20) | 处理状态，pending、processed |
| reply | text | 管理员回复 |
| create_time | datetime | 创建时间 |
| update_time | datetime | 更新时间 |

建议索引：

- `user_id` 普通索引
- `status` 普通索引

## 四、可选扩展表

### 1. 情绪报告表 emotion_report

如果后续需要保存周报和月报，可以增加该表。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 报告 ID，主键 |
| user_id | bigint | 用户 ID |
| report_type | varchar(20) | 报告类型，week 或 month |
| start_date | date | 统计开始日期 |
| end_date | date | 统计结束日期 |
| main_emotion | varchar(50) | 主要情绪 |
| keywords | varchar(255) | 高频关键词 |
| summary | text | 报告总结 |
| suggestion | text | AI 建议 |
| create_time | datetime | 创建时间 |

说明：

- 基础版可以不建该表，周报和月报由后端实时生成。
- 如果需要查看历史报告，再考虑保存报告数据。

## 五、表关系说明

| 关系 | 说明 |
| --- | --- |
| user 1 - N emotion_diary | 一个用户可以有多篇情绪日记 |
| user 1 - N ai_chat_message | 一个用户可以有多条 AI 对话消息 |
| user 1 - N feedback | 一个用户可以提交多条反馈 |
| user 1 - N emotion_report | 一个用户可以生成多个周报或月报 |

## 六、统计数据来源

情绪趋势、情绪分布、情绪强度统计和关键词统计可以先从 `emotion_diary` 表查询得到。

常见统计方式：

- 按日期统计平均情绪分数，用于趋势折线图。
- 按情绪极性统计数量，用于积极、消极、中性占比图。
- 按情绪类型统计数量，用于情绪分布图。
- 汇总关键词字段，用于高频关键词展示。

## 七、安全与隐私说明

- 密码必须加密存储，不能保存明文密码。
- 普通用户只能查询、修改和删除自己的数据。
- 管理员查看日记数据时，建议只显示摘要或统计结果。
- 论文和演示中应使用测试数据，不使用真实隐私数据。
- 与风险表达相关的数据只用于安全提示，不用于医疗判断。

## 八、后续调整方向

- 根据实际接口增加字段校验规则。
- 根据查询速度增加必要索引。
- 如果 AI 调用记录需要审计，可以增加 `ai_call_log` 表。
- 如果统计查询变慢，可以增加单独的统计缓存表。
- 如果需要多轮对话会话管理，可以增加 `ai_chat_session` 表。
