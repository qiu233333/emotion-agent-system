# 接口设计初稿

## 一、接口设计说明

本系统采用前后端分离架构，前端通过 Axios 调用后端 RESTful API。后端使用 Spring Boot 提供接口，并通过 JWT 完成用户身份认证。

接口路径暂定以 `/api` 开头，后续开发时可以根据实际代码进行微调。

## 二、统一返回格式

后端接口建议使用统一返回格式，方便前端统一处理成功、失败和错误提示。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| code | int | 状态码，200 表示成功 |
| message | string | 提示信息 |
| data | object | 返回数据 |

示例结构：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

## 三、认证方式

登录成功后，后端返回 JWT Token。前端在后续请求中通过请求头携带 Token。

| 请求头 | 说明 |
| --- | --- |
| Authorization | Bearer token |

不需要登录的接口：

- 用户注册
- 用户登录

需要登录的接口：

- 用户信息
- 情绪日记
- AI 分析
- 情绪统计
- 用户反馈

管理员接口需要管理员角色权限。

| 参数 | 类型 | 必填 | 说明 |

## 五、情绪日记接口

| 接口 | 方法 | 是否登录 | 功能 |
| --- | --- | --- | --- |
| `/api/diary` | POST | 是 | 新增情绪日记 |
| `/api/diary/list` | GET | 是 | 查询日记列表 |
| `/api/diary/{id}` | GET | 是 | 查询日记详情 |
| `/api/diary/{id}` | PUT | 是 | 修改日记 |
| `/api/diary/{id}` | DELETE | 是 | 删除日记 |

### 1. 新增情绪日记

请求地址：`POST /api/diary`

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| title | string | 是 | 日记标题 |
| content | string | 是 | 日记内容 |
| moodTag | string | 否 | 用户自选心情标签 |
| diaryDate | date | 是 | 日记日期 |

处理流程：

1. 后端接收日记内容。
2. 保存基础日记信息。
3. 调用情绪分析模块。
4. 调用大语言模型生成建议。
5. 保存分析结果和 AI 建议。
6. 返回完整日记详情。

### 2. 查询日记列表

请求地址：`GET /api/diary/list`

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| page | int | 否 | 当前页 |
| size | int | 否 | 每页条数 |
| startDate | date | 否 | 开始日期 |
| endDate | date | 否 | 结束日期 |
| emotionType | string | 否 | 情绪类型 |

返回数据：分页日记列表。

### 3. 查询日记详情

请求地址：`GET /api/diary/{id}`

返回数据：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | long | 日记 ID |
| title | string | 日记标题 |
| content | string | 日记内容 |
| moodTag | string | 心情标签 |
| emotionPolarity | string | 情绪极性 |
| emotionType | string | 情绪类型 |
| emotionScore | int | 情绪强度 |
| keywords | string | 关键词 |
| aiSummary | string | AI 分析摘要 |
| aiSuggestion | string | AI 建议 |
| diaryDate | date | 日记日期 |

## 十、接口设计注意事项

- 用户只能访问自己的日记、统计数据和对话记录。
- 管理员查看日记数据时，应尽量只展示统计信息或脱敏内容。
- AI 相关接口需要处理超时、调用失败和返回为空的情况。
- 对包含极端负面表达的文本，应返回安全提示，不输出危险建议。
- 列表接口需要支持分页，避免一次性查询过多数据。
